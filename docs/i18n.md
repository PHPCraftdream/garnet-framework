# I18n (Internationalization)

## Overview

Garnet uses a code-generation approach for i18n. Translations are
defined in PHP source files and a build step converts them into
TypeScript shims for the frontend. Both sides read the same keys
through the same `t.Key()` shape, so a missing key is a compile-time
error on the TS side and a runtime exception on the PHP side.

## Source files

PHP translation files live in each bundle's `I18n/` directory:

```
Framework/Bundle/I18n/
├── FwI18n.php           # the I18n class that loads translations
├── I18nDataEn.php       # English translations
└── I18nDataRu.php       # Russian translations
```

Each `I18nData*.php` returns a `$data` array of key → value pairs:

```php
<?php
namespace PHPCraftdream\Garnet\Bundle\I18n;

class I18nDataEn
{
    public static array $data = [
        'Common_Email' => 'Must be a valid email',
        'Common_Int'   => 'Must be an integer',
        'Common_Min'   => 'Minimum value: %s',
        // ...
    ];
}
```

## Generation

### How it works

1. **Trigger:** `PrepareParams` calls `$appInfo->dumpFrontLang()` for every bundle.
2. **Generator:** `Kernel/Io/I18n/I18nFrontendDumper.php`.
3. **Template:** `Templates/CodeFiles/TrTs.template`.

### Generated files

For each bundle, TypeScript shims land under
`<bundle>/Front/I18nGen/`:

```
Framework/Bundle/Front/I18nGen/
├── I18nDataEN.ts       # static English translations
├── I18nDataRU.ts       # static Russian translations
└── I18nFramework.ts    # the typed shim class
```

The generated `I18nFramework.ts` provides type-safe access:

```typescript
import { I18nFramework } from '@common/I18n/I18nFramework';

const msg = I18nFramework.Common_Email();     // typed string
const min = I18nFramework.Common_Min([5]);    // with interpolation
```

## Commands

### Regenerate the i18n files

```bash
php garnet prepare
```

`prepare` re-runs the dumper for every bundle. Any change to a
`I18nData*.php` requires `prepare` before the new key is visible on
the TS side.

### Rebuild the frontend

```bash
php garnet build
```

The build is content-hash-aware: re-running `prepare` rewrites the TS
shims, which forces the bundle chunk hashes to refresh.

## Adding a translation

1. **Edit both language files** in the bundle's `I18n/`:
   - `…I18nDataRu.php` (Russian)
   - `…I18nDataEn.php` (English)
2. **Add the same key** to both arrays:
   ```php
   'Validation_CustomRule' => 'Custom validation failed',
   ```
3. **Regenerate**:
   ```bash
   php garnet prepare
   ```
4. **Use it** in PHP or TS:
   ```php
   $t = FwI18n::getInstance();
   $msg = $t->Validation_CustomRule();
   ```
   ```typescript
   const msg = I18nFramework.Validation_CustomRule();
   ```

## Bundle-specific i18n

Every bundle has its own `I18n/` directory and produces its own TS
class. For example an app bundle:

```
Apps/MyApp/Foreground/I18n/
├── ForegroundI18n.php
├── ForegroundI18nDataEn.php
└── ForegroundI18nDataRu.php
```

Generated to:

```
Apps/MyApp/Foreground/I18nGen/
├── I18nDataEN.ts
├── I18nDataRU.ts
└── I18nForeground.ts
```

## Important

- **Never edit the generated TypeScript files.** They are overwritten
  on every `prepare`.
- The `I18nGen/` folder name marks generated content (Gen = generated).
- Type safety: the generated class autocompletes every key in your IDE.
- Interpolation: use `%s` placeholders in translations and pass
  arguments to the call: `I18nFramework.Common_Min([5])`.

## Variable interpolation (`%s`)

Translation strings use `%s` placeholders. The engine substitutes
them — **never wrap the call in `sprintf()` yourself**.

### PHP

`FwI18n::tr()` and the `__call` magic method already do
`sprintf($tr, ...$args)` internally. Pass arguments directly to the
generated method in the same order as the `%s` placeholders:

```php
// Translations:
//   'Email_BookingCreated_Subject' => 'New booking on %s',
//   'Common_Min'                   => 'Minimum: %s',
//   'Common_Range'                 => 'From %s to %s',

// CORRECT — pass args to the i18n method:
$subject = $t->Email_BookingCreated_Subject('My Brand');
$msg     = $t->Common_Min(5);
$range   = $t->Common_Range(1, 10);

// WRONG — double substitution / wasted call:
$subject = sprintf($t->Email_BookingCreated_Subject(), 'My Brand');  // ❌
```

If a string has no placeholders, call without args:

```php
$title = $t->Email_BookingCreated_Title();
```

### TypeScript

The generated frontend class takes an **array** of args:

```typescript
import { I18nFramework } from '@common/I18n/I18nFramework';

const min   = I18nFramework.Common_Min([5]);
const range = I18nFramework.Common_Range([1, 10]);
const title = I18nFramework.Email_BookingCreated_Title();  // no args needed
```

## Brand name

For strings that reference the application's brand name (e.g. email
subjects like _"Booking confirmed on My Brand"_), don't hard-code the
brand. Use a `%s` placeholder and pass `FwAppSettings::brandName()` —
the framework-level helper that reads `title` from `app.ini` with a
fallback.

```php
use PHPCraftdream\Garnet\Bundle\Modules\SystemSettings\FwAppSettings;

// Translation: 'Email_BookingCreated_Subject' => 'New booking on %s'
$subject = $t->Email_BookingCreated_Subject(FwAppSettings::brandName());
```

This keeps the brand a single source of truth (the `title` key in
`app.ini`) and lets the same translation file ship across white-label
deployments without code changes.
